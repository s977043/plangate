# PlanGate 自己成長・自己進化に関する Claude Code × Codex ディスカッション

- 日時: 2026-05-14
- 主題: Steering Loop 以外で PlanGate を自己成長・自己進化させる強化軸を検証する
- 参加者: Claude Code（Opus 4.7）/ Codex CLI
- 前提: 同日の Steering Loop ディスカッション（`2026-05-14-steering-loop-codex.md`）で Control Plane track を既存 Roadmap に重ねる方針が確定済み

## 問題意識
Steering Loop（観測 → 判定 → 制御の決定論的ループ）は強化軸の 1 つに過ぎない可能性がある。Steering Loop 偏重に陥らず、自己成長・自己進化の手段を網羅検証したい。


## Round 1 — Claude → Codex: 強化軸の網羅列挙

### Claude のプロンプト（要旨）
自己進化のための強化軸として 7 軸を仮説提示。漏れている軸の追加、各軸の本質/接点/難度/時間軸の整理、Steering Loop 中心主義への反論、階層構造の明示を依頼。

### Codex の回答全文

**結論**
7軸はかなり妥当です。ただし、自己進化システムとして見ると、少なくとも次の軸が不足しています。

- **観測軸**: 何が起きたかを測れる状態にする
- **因果分析軸**: なぜ成功・失敗したかを説明可能にする
- **ガバナンス軸**: 改善候補を誰が、どの条件で採用するかを制御する
- **安全性軸**: 自己改善が品質・互換性・セキュリティを壊さないようにする
- **経済性軸**: 改善活動のコスト、時間、認知負荷を最適化する
- **標準化軸**: 良い判断や成果物を再利用可能な型にする
- **外部環境適応軸**: モデル、CLI、ツール、GitHub Actions など外部変化に追従する

また、あなたの7軸のうち一部は独立軸というより上位軸の実装形です。

- `記憶軸` は独立でよい。ただし Steering Loop の一部ではなく、観測・学習・適応の基盤。
- `品質軸` は自己評価軸に併合可能。ただし「過去TASK再走」は重要なので、自己評価軸の中核機能として残すべき。
- `進化軸` は広すぎるため、A/B・bandit は「探索最適化軸」、採用判断は「ガバナンス軸」に分ける方がよい。
- `協調軸` は独立でよいが、常設レビューだけでなく「意見差分を意思決定材料に変換する」軸として扱うべき。
- `Steering Loop` は観測軸・因果分析軸・再現性軸の基盤ではあるが、自己進化全体の中心とは限らない。

| 3. 学習軸 | retrospective や過去TASKから、反復可能な playbook、チェックリスト、スキル改善候補を抽出する。経験を属人的メモから運用資産へ変える。 | `AGENT_LEARNINGS.md`、`.agents/skills/`、handoff、docs/working | 中 | スプリント |

| 6. 探索最適化軸 | prompt、profile、agent role、gate順序、レビュー観点の候補を比較し、より良い運用設定を探す。A/B や bandit はこの軸。 | `.codex/agents/*.toml`、`.codex/config.toml`、skills、eval-runner | 極高 | 四半期 |

| 8. 記憶軸 | 過去の判断、失敗、採用理由、却下理由を検索・再利用可能にする。単なるログではなく、意思決定履歴として扱う。 | `AGENT_LEARNINGS.md`、docs/working、discussion logs、handoff | 中 | スプリント |
| 9. 適応軸 | ユーザー、リポジトリ、タスク種別ごとの好み・制約・成功パターンに運用を合わせる。mode 自動推定もここに入る。 | project-rules、core-contract、working-context、mode判定 | 高 | 四半期 |
| 10. 因果分析軸 | 成功・失敗の原因を、単なる結果ログではなく「どの判断が影響したか」として説明する。自己進化で最も抜けやすい軸。 | gate結果、review comments、handoff、Trace Timeline | 高 | 四半期 |
| 11. ガバナンス軸 | 改善候補をいつ採用し、いつ保留し、いつロールバックするかを決める。自己進化を「勝手に変わる」ではなく「人間が判断できる改善パイプライン」にする。 | issue-governance、milestone、PBI、C/V gates | 中 | 即時〜スプリント |
| 12. 安全性軸 | ハーネス改善が既存契約、後方互換、秘密情報、破壊的操作を壊さないようにする。自己改善のブレーキ。 | core-contract、Stop rules、EH-8 privacy、後方互換制約 | 中 | 即時 |
| 13. 経済性軸 | 改善のためのトークン、時間、手順、レビュー負荷を測り、1人運用で維持できる範囲に抑える。自己進化は運用コストで死にやすい。 | Progressive Disclosure、mode、gate選択、scripts | 中 | スプリント |
| 14. 標準化軸 | 良い成果物や判断をテンプレート、skill、checklist、schema に昇格させる。再利用可能な型を増やす軸。 | `.agents/skills/`、templates、docs/ai、PBI artifacts | 低〜中 | 即時 |
| 15. 外部環境適応軸 | モデル仕様、CLI挙動、GitHub API、CI、外部ツールの変化に追従する。PlanGate 自体では制御できない変化への耐性。 | Codex/Claude設定、tool roles、release notes、runtime検証 | 高 | 四半期 |

**Steering Loop 中心主義への反論**
Steering Loop は強い基盤ですが、自己成長の唯一の入口ではありません。特に次の条件では後回しでもよいです。

1. **まだ改善対象が曖昧な場合**  
   先に学習軸・標準化軸を進める方がよいです。retrospective から「何を毎回間違えるか」を抽出しないまま制御ログだけ厚くしても、記録対象が膨らむだけです。

2. **1人運用の認知負荷が限界に近い場合**  
   経済性軸とガバナンス軸を先に置くべきです。Steering Loop は観測量を増やすため、読む・判断する・保守する負荷も増えます。

3. **直近の痛みが成果物品質にある場合**  
   自己評価軸、品質回帰軸を先にする方が筋がよいです。過去TASK再走や acceptance-review の標準化は、Steering Loop が未完成でも効果が出ます。

4. **既存資産がまだ skill / checklist に昇格していない場合**  
   標準化軸が先です。良い判断が毎回ログに残っても、それが次回の手順に入らなければ自己進化になりません。

5. **安全・後方互換の契約が弱い場合**  
   安全性軸とガバナンス軸を先にすべきです。自己進化の前に「変えてはいけないもの」「人間承認が必要なもの」を明文化する必要があります。

つまり、Steering Loop は **自己進化の観測・再現・因果分析を強くする装置** ですが、最初に必要なのは必ずしもそこではありません。短期では「標準化」「自己評価」「ガバナンス」の方が効く場面があります。

**階層構造**
フラットではありません。大きく分けると、次の順序です。

```text
Level 0: 制約・安全
- 安全性軸
- ガバナンス軸
- 経済性軸

Level 1: 見える化
- 観測軸
- 記憶軸
- 外部環境適応軸

Level 2: 比較・説明
- 再現性軸
- 自己評価軸
- 品質回帰軸
- 因果分析軸

Level 3: 改善候補生成
- 学習軸
- 協調軸
- 標準化軸

Level 4: 改善最適化
- 適応軸
- 探索最適化軸
```

依存関係で見るとこうです。

```text
安全性・ガバナンス・経済性
        ↓
観測・記憶
        ↓
再現性・自己評価・品質回帰
        ↓
因果分析
        ↓
学習・協調・標準化
        ↓
適応・探索最適化
```

ただし実行順はこの通りでなくてよいです。1人運用なら、現実的には次の順が軽いです。

1. **標準化軸**: よくある成功・失敗を skill/checklist 化する  
2. **自己評価軸**: 過去TASK再走の最小セットを作る  
3. **ガバナンス軸**: 改善候補の採用・保留・破棄ルールを作る  
4. **観測軸 / Steering Loop**: 制御点を構造化して残す  
5. **因果分析軸**: どの判断が結果に効いたかを追う  
6. **探索最適化軸**: A/B、bandit、自動探索に進む  

**監査コメント**
Steering Loop を「自己進化の中心」と呼ぶとやや危険です。正確には、Steering Loop は **自己進化のための観測・再現基盤** です。

自己進化の中心はむしろ次の3つです。

- **評価**: 何が良くなったと言えるのか
- **学習**: その知見を次回の行動にどう反映するのか
- **ガバナンス**: 反映してよいかを誰がどう判断するのか

Steering Loop はこの3つを強化しますが、代替しません。PlanGate の自己成長ロードマップでは、Steering Loop を上位目的にせず、`評価 → 学習 → ガバナンス` を上位に置き、その下に `観測・再現・因果分析` として配置するのが健全です。
`brainstorming` スキルの前提に従い、今回は実装案ではなく「軸の地図」として整理します。

自己進化の中心はむしろ次の3つです。

- **評価**: 何が良くなったと言えるのか
- **学習**: その知見を次回の行動にどう反映するのか
- **ガバナンス**: 反映してよいかを誰がどう判断するのか

Steering Loop はこの3つを強化しますが、代替しません。PlanGate の自己成長ロードマップでは、Steering Loop を上位目的にせず、`評価 → 学習 → ガバナンス` を上位に置き、その下に `観測・再現・因果分析` として配置するのが健全です。

## Round 2 — Claude → Codex: 着手順序の確定

### Claude のプロンプト（要旨）
PlanGate の既存資産バランス（標準化が既に高い、ガバナンスも高い、観測/評価/学習が低い）を踏まえ、Codex 推奨順序の再評価、2 番目に着手すべき軸の確定、複利が最も効く組み合わせ、半年ロードマップへの落とし込みを依頼。

### Codex の回答全文

**Q5. Codex 推奨順序の再評価**

推奨: トップ 3 は **自己評価 → 学習 → ガバナンス実装** に組み直す。

理由: 「標準化 → 自己評価」という順序は、標準化が未整備なプロジェクトでは正しい。ただし現在の PlanGate は、スキル、PBI、handoff、issue governance、ゲート設計がすでに高い水準にあるため、標準化をさらに先行させる投資効率は下がっている。

いま不足しているのは「標準があること」ではなく、標準が実行結果を改善しているかを測る仕組み。したがって最初に必要なのは **PlanGate 自身を評価対象にする自己評価軸**。次に、その評価結果を再利用可能な改善知に変える **学習軸**。その後に、学習された改善を逸脱なく運用へ組み込む **ガバナンス実装** が来る。

再評価後の現実的順序:

1. **自己評価**: dogfooding eval suite
2. **学習**: retrospective → playbook 昇格導線
3. **ガバナンス実装**: Orchestrator Mode 不変条件の Hook 化

標準化は「次に着手する軸」ではなく、これらを載せる既存基盤として扱うのが妥当。

**Q6. 2 番目に着手すべき軸**

推奨: **(B) 自己評価軸: PlanGate dogfooding eval suite**

理由: Trace Timeline v1 / Gate Event Normalization を 1 番目とするなら、2 番目はその観測データを使って「良い実行だったか」を判定する軸にするべき。観測だけでは改善は起きず、評価基準がないとログは増えるだけになる。

学習軸も重要だが、評価なしに retrospective → playbook 自動化へ進むと、何を昇格すべきかの判定が主観に寄る。品質回帰ベンチマークも有力だが、まず現在の PlanGate 実行を評価できる最小 suite がないと、過去 TASK 再走の合否基準も曖昧になる。

したがって 2 番目は自己評価。具体的には、まず以下のような小さな dogfooding eval から始めるのがよい。

- PBI 入力から AC / Design / Task 分解が妥当か
- handoff 6 要素が満たされているか
- C-3 / C-4 判定に必要な証跡があるか
- Trace Timeline に評価可能なイベントが揃っているか
- Stop rules / core-contract 違反がないか

これは既存の標準化資産をそのまま評価項目に変換できるため、実装効率も高い。

**Q7. 自己進化を最も加速する組み合わせ**

推奨: **Trace Timeline v1 × 自己評価軸**

理由: 最も複利が効くのは、観測データが評価入力になり、評価結果が次の観測設計を改善するループだから。

流れはこうなる。

1. Trace Timeline が実行過程を構造化して残す
2. dogfooding eval suite がその Timeline を評価する
3. 評価結果から「足りないイベント」「曖昧な判定」「弱い handoff」が見える
4. 次の Trace Timeline / schema / gate event が改善される
5. 改善された観測が、さらに精度の高い自己評価を可能にする

この循環は、単なるログ蓄積でも、単なる評価表でもない。観測の粒度と評価の精度が相互に上がるため、自己進化の土台として複利が強い。

学習軸との組み合わせも最終的には重要。ただし学習は、評価済みの成功・失敗事例が増えてからの方が質が上がる。最初の複利エンジンは **観測 → 評価** に置くべき。

**Q8. 半年ロードマップへの落とし込み**

推奨: **v8.7.0 観測・再現性 → v8.8.0 自己評価 → v8.9.0 学習**

各 milestone で 1 軸ずつ積むなら、この順序が最も筋がよい。

v8.7.0: **観測・再現性**
Trace Timeline v1 / Gate Event Normalization を入れる。目的は、PlanGate 実行を後から説明・比較・評価できる形で残すこと。ここでは「改善」まで欲張らず、評価可能な証跡を作る。

PlanGate dogfooding eval suite を入れる。v8.7.0 の Timeline / gate events / handoff / decision log を入力にして、実行品質を判定する。ここで初めて「PlanGate が PlanGate 自身を評価する」状態になる。

v8.9.0: **学習**
retrospective → playbook / skill / rule 昇格導線を作る。v8.8.0 の評価結果をもとに、成功パターン・失敗パターンを再利用可能な知識へ変換する。ここで自己進化の中心である「評価 → 学習 → ガバナンス」のうち、評価と学習が接続される。

結論として、半年の主軸は以下。

```text
v8.7.0: 観測・再現性
v8.8.0: 自己評価
v8.9.0: 学習
```

ガバナンス Hook 実装は重要だが、v8.9.0 の後、または v8.9.0 内の副次タスクとして扱うのがよい。評価・学習の出力がない段階で Hook 化を進めると、守るべき不変条件は増えるが、改善サイクルはまだ太くならない。

## 決定事項サマリ（Claude 統合）

### 重要な認識転換
- **Steering Loop は自己進化の "中心" ではなく "観測・再現基盤"**
- 自己進化の真の中心は **「評価 → 学習 → ガバナンス」**
- 既存 PlanGate は標準化・ガバナンス資産が既に充実。不足は「**評価・学習・因果分析**」

### 自己進化 15 軸の階層
```
Level 0 (制約・安全):   安全性 / ガバナンス / 経済性
Level 1 (見える化):     観測 / 記憶 / 外部環境適応
Level 2 (比較・説明):   再現性 / 自己評価 / 品質回帰 / 因果分析
Level 3 (改善候補生成): 学習 / 協調 / 標準化
Level 4 (改善最適化):   適応 / 探索最適化
```

### 半年ロードマップ（軸ベース）
| Milestone | 主軸 | 主成果物 |
|----------|-----|---------|
| v8.7.0 | 観測・再現性 | Trace Timeline v1 / Gate Event Normalization（既決） |
| v8.8.0 | **自己評価** | PlanGate dogfooding eval suite |
| v8.9.0 | 学習 | retrospective → playbook/skill/rule 昇格導線 |
| 後続 | ガバナンス実装 / 因果分析 / 探索最適化 | Orchestrator 不変条件 Hook 化など |

### 複利エンジンの構造
```
Trace Timeline (観測)
  ↓ 評価入力
Dogfooding Eval (自己評価)
  ↓ 不足検出
観測設計の改善 → さらに精度の高い評価
  ↓ 評価済み事例蓄積
playbook 昇格 (学習)
  ↓ 改善知の再利用
```
観測の粒度と評価の精度が相互に上がるループが自己進化の土台。

### Codex の監査コメント（重要）
> Steering Loop を「自己進化の中心」と呼ぶとやや危険。正確には自己進化のための **観測・再現基盤**。中心は「評価・学習・ガバナンス」。Steering Loop はこの 3 つを強化するが代替しない。
