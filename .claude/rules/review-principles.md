# レビュー原則（CI/ローカル共通）

CI自動レビュー・ローカルレビュー共通の判定フレーム。

## 1. 三相レビューモデル

| Phase | 名称 | 焦点 | 対応する観点 |
|-------|------|------|-------------|
| Upstream | 設計判定 | アーキテクチャ、依存方向、責務分離 | 拡張性、保守性 |
| Midstream | 実装判定 | コード品質、ロジック正確性、安全性 | 可読性、パフォーマンス、セキュリティ |
| Downstream | テスト判定 | テスト品質、カバレッジ、パターン準拠 | 保守性 |

## 2. 5つのレビュー観点

1. **可読性(Readability)**: 命名の適切さ、構造の明確さ、コードの理解しやすさ
2. **拡張性(Extensibility)**: アーキテクチャ境界、依存方向、責務分離
3. **パフォーマンス(Performance)**: N+1クエリ、ループ内I/O、不要データ取得、リソース効率
4. **セキュリティ(Security)**: 入力バリデーション、認証・認可、機密情報保護、インジェクション対策
5. **保守性(Maintainability)**: テスト品質、既存パターン準拠、デバッグのしやすさ、残骸の有無

## 3. Severity定義（4段階）

| Severity | 定義 | 例 | マージ影響 |
|----------|------|---|-----------|
| **critical** | 本番障害・データ不整合・脆弱性 | SQLインジェクション、認可チェック漏れ、データ損失 | ブロッカー |
| **major** | ロジック誤り・テスト不足・設計違反 | N+1クエリ、レイヤー間依存違反、重要パス未テスト | 修正推奨 |
| **minor** | 改善提案・命名・コードスタイル | 変数名改善、early return提案、ドキュメント不備 | 任意 |
| **info** | FYI・将来課題・好みの問題 | 類似パターンの紹介、リファクター候補、補足情報 | 無視可 |

## 4. 判定基準

| 判定 | 条件 |
|------|------|
| **Auto-approve** | critical=0、major=0、セキュリティ観点で問題なし |
| **Human review recommended** | major >= 1 または 総合スコア70-89 |
| **Human review required** | critical >= 1 または 総合スコア < 70 |

## 5. レビューの姿勢

- **具体性を重視**: 一般論ではなく、差分に基づいた具体的なコメントを提供する
- **改善案の提示**: 問題を指摘するだけでなく、可能な場合は改善案や代替案も提示する
- **建設的なトーン**: 開発者を支援することを目的とし、中立的・協力的な口調を保つ
- **故障確率で判断**: 「好み」ではなく「故障確率」で指摘する

## 6. 禁止事項

- **過度な推測**: 差分に存在しないコードへの推測に基づいた指摘
- **抽象的なレビュー**: 一般論だけのレビュー（具体的な差分への言及なし）
- **範囲外の指摘**: 変更されていないコードへの過度なレビュー（セキュリティは例外）
- **不適切なトーン**: 批判的・攻撃的な口調、人格や能力への攻撃

## 7. False-positiveガード

以下は指摘しない:

- 静的解析ツール（linter、型チェッカー等）で自動検出可能な問題
- nit-pickingの過多（minor/infoは合計5件以内に絞る）
- 変更範囲外のコードの問題（ただしsecurityは例外）
- 既存コードのスタイル統一（今回の変更で新たに導入された場合のみ指摘）
- 根拠のない仮定に基づくレビュー（推測は断定しない）

## 8. レビューの優先順位

1. **Critical**: セキュリティ脆弱性、データ損失のリスク、システムダウンの可能性
2. **Major**: 重大なバグ、パフォーマンス問題、設計上の大きな問題
3. **Minor**: 小さなバグ、可読性の問題、軽微な最適化の機会
4. **Info**: 提案、参考情報、追加の検討事項
